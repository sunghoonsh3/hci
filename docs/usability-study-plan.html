Usability Study Plan: PATH (Registration Clarity)

Overview

PATH is a browser-based rewrite of the course-registration workflow at Notre Dame. Students currently bounce between a catalog search (PATH) and a separate registration form (NOVO), and most of the frustration happens because prerequisites, seat limits, and time conflicts only surface after a failed submission. Our version keeps the two tools on one screen. A student pastes their Degree Works audit, searches the Summer 2026 catalog, and sees an eligibility badge on every row. Clicking into a course shows the prereq text rewritten in plain English, with green checks for requirements the student has already met and red marks for the ones they have not. Students build up to three schedule drafts side by side in a Plan A / B / C view, where a weekly calendar flags any time conflict. Before exporting to NOVO, a pre-check runs every rule the registrar would run, so the student catches problems the night before, not at 7 a.m. on registration morning.

Participants

Five Notre Dame undergraduates, each registering for Summer or Fall 2026. Five is a common size for formative testing, and it will catch the painful issues, but it is obviously not a representative sample of the whole student body. What we can do is spread the five across the dimensions most likely to change how someone uses the tool.

Tasks

Every participant works off the same seeded persona: Alex Murphy, a senior Computer Science major whose audit is already in the system. The tasks are printed on an index card and read out loud at the start of each one.

Imagine you are Alex Murphy. The text on the card is your Degree Works audit. Get it into PATH and then point to where the system shows your major, your class standing, and the CSE courses you have already finished.

Alex still owes the university a theology course. Pull up THEO 10001: Foundations of Theology: Biblical/Historical for Summer 2026 and tell me whether Alex can sign up. If something would stop him, say what.

Use Plan A, Plan B, and Plan C to sketch three different Summer 2026 schedules for Alex. Each one needs at least two courses, at least six credits, and no time conflicts on the calendar. They cannot all be the same schedule.

Add CSE 30872: Programming Challenges to Plan A. You will get a warning because Alex has already taken it. Decide how you want to handle that, using only what PATH is telling you, and explain what you did.

Open the Export to NOVO screen. Walk through the pre-check on Plan A, fix whatever it calls out, and then export.

Measures

Three categories, each with one thing we measure from the screen recording and one thing we ask the participant.

CategoryObjective (from recording)Subjective (from the participant)
Effectiveness Success, partial, or failure on each task, scored against a written rule written before the study starts. Example rule for Task 3: three plans exist, each with ≥ 6 credits, no conflicts, not identical. We separately count critical errors, meaning actions that would have gone through as a real but wrong registration, such as exporting a plan that still had a conflict. After each task, a single 7-point item: “I’m confident I did that task correctly.” A participant who scores a 7 but actually failed is the most useful data point we can collect, because it means PATH let them walk out feeling fine about a broken schedule.
Efficiency Wall-clock time on task and total click count, pulled from the recording. Reported as a median and range across the five participants, not a mean, because five data points make means misleading. SMEQ after each task. One vertical 0–150 scale with verbal anchors from “Not at all hard to do” to “Tremendously hard to do.” Takes about ten seconds to fill in, so it does not derail momentum between tasks.
Satisfaction Reaction coding. Two of us independently watch the session tapes and tally positive moments (smile, nod, “oh nice”) and negative moments (sigh, eyebrow raise, leaning back, audible swear) against each task. We report counts plus Cohen’s kappa so readers can judge how reliable the coding is. System Usability Scale at the end of the session. Ten items, 1–5. Scoring and interpretation are in Appendix A. We want PATH above 72, and we would not ship anything below 68.

A short exit interview follows: most frustrating moment, easiest moment, one thing you would change.

Procedure

Each session is about 75 minutes in a group-study room at Hesburgh. The setup is one MacBook running the PATH build, OBS for screen and microphone, and a camcorder on a tripod aimed at the participant’s hands and face.

The facilitator opens with the usual test-the-software-not-the-person line, gets the consent form and recording release signed, and collects a small demographic sheet (year, college, major, 1–5 rating of comfort with the current PATH and NOVO tools).

Before the real tasks, we run a two-minute think-aloud warm-up on a neutral site, usually a flight search, so narrating thoughts stops feeling weird. Then the five tasks, in order. Between tasks the facilitator stops the timer, resets the app to a clean seeded state, and hands over the confidence item and the SMEQ sheet. If someone is clearly stuck and asks for help, the only response is “What are you trying to do right now?” Nothing beyond that.

After Task 5, the participant fills out the SUS on paper, with the browser closed so they are not half-looking at the interface while they answer. Then the exit interview, a thank-you, and the gift card.

Analysis happens offline. Two researchers code the tapes separately for task outcome, click count, and reaction events, and meet to reconcile. Quantitative results land in a one-page summary. The interview notes and coded frustrations get sorted into an affinity diagram, which becomes the next sprint’s backlog.

Appendix A: System Usability Scale

Respondents rate each statement 1 (strongly disagree) to 5 (strongly agree).

  1. I think I would like to use this system frequently.
  2. I found the system unnecessarily complex.
  3. I thought the system was easy to use.
  4. I think I would need the support of a technical person to be able to use this system.
  5. I found the various functions in this system were well integrated.
  6. I thought there was too much inconsistency in this system.
  7. I would imagine most people would learn to use this system very quickly.
  8. I found the system very cumbersome to use.
  9. I felt very confident using the system.
  10. I needed to learn a lot of things before I could get going with this system.

Scoring: for odd items, subtract 1 from the response. For even items, subtract the response from 5. Add the ten numbers and multiply by 2.5. The result is a 0–100 score. Above 80 is excellent, 68 is the industry average, 51 to 68 is marginal, and below 51 is unacceptable.
